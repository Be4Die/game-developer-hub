## Проект

**«Центр разработчиков»** — система управления публикацией и сопровождением веб-игр
на платформе Welwise Games. Дипломная работа (ВКР), выполняемая на заказ.

Полное ТЗ: `docs/tech-spec.pdf`

## Цель взаимодействия

Автор проекта **учится писать на Go**. Главная задача — не просто получить рабочий код,
а вырастить из автора уверенного Go-разработчика уровня junior+.

### Как помогать

- **Объясняй «почему», а не только «как».** Каждое архитектурное решение, паттерн или
  идиому Go сопровождай кратким объяснением мотивации.
- **Давай код порциями.** Не выдавай готовый файл на 300 строк — разбивай на шаги,
  чтобы автор понимал каждый блок.
- **Указывай на идиоматичность.** Если автор пишет «не по-гошному», мягко поправляй
  и показывай идиоматичный вариант.
- **Предлагай самостоятельные упражнения.** После демонстрации паттерна предложи
  автору написать аналогичный код самому.
- **Не прячь ошибки.** Если в предложенном автором коде есть проблема — объясни, в чём
  она, до того как давать исправленный вариант.

## Стек и соглашения

| Слой | Технология |
|---|---|
| Язык бэкенда | Go 1.24+ |
| Транспорт (между сервисами) | gRPC + Protobuf (buf) |
| Транспорт (клиент ↔ API) | REST / JSON (планируется) |
| БД | PostgreSQL |
| Фронтенд | Vue.js (SPA) |
| Контейнеризация | Docker |
| Задачи / скрипты | Taskfile (`Taskfile.yml`) |

### Структура Go-сервисов (шаблон)

```
service-name/
├── cmd/<binary>/main.go   # точка входа, минимум логики
├── config/                # YAML-конфигурации по окружениям
├── internal/
│   ├── app/               # сборка зависимостей (wire / ручной DI)
│   ├── config/            # парсинг конфигурации
│   ├── domain/            # доменные модели, интерфейсы, ошибки
│   ├── service/           # бизнес-логика (use-cases)
│   ├── storage/           # реализации репозиториев (memory, postgres)
│   └── transport/         # gRPC / HTTP хэндлеры, конвертеры
└── Dockerfile
```

### Стиль кода

- **Go conventions**: `gofmt`, `go vet`, именование по Effective Go.
- Ошибки — через `fmt.Errorf("operation: %w", err)` с оборачиванием.
- Доменные ошибки — sentinel-переменные в `domain/errors.go`.
- Логгер — `log/slog` (structured logging).
- Конфигурация — `cleanenv` + YAML.
- Зависимости в сервисах — через интерфейсы (dependency inversion).
- **Никаких `init()` и глобального состояния.**

## Подсистемы (из ТЗ)

1. Управление проектами — CRUD черновиков, загрузка билдов, метаданные, доступ.
2. Развёртывание — автодеплой билдов в staging/production окружения.
3. Модерация — тикеты, вердикты, встроенный чат.
4. Оркестрация серверов — загрузка серверных билдов, запуск/остановка инстансов,
   логи, service discovery, подключение собственных нод разработчика.
5. Аналитика и отчётность — сбор метрик, графики, выгрузка отчётов.
6. Авторизация — регистрация, логин, восстановление пароля, роли.

Сейчас в активной разработке: **game-server-node** (подсистема оркестрации, агент ноды).

## Текущее состояние репозитория

- `game-server-node/` — Go-сервис агента вычислительного узла (gRPC transport,
  in-memory storage, domain models для инстансов и ресурсов).
- `protos/` — Protobuf-определения (`game_server_node/v1`: deployment, discovery).
- `docs/` — ТЗ, диаграммы, заметки.

## Контекст при ответах

- OS автора: **Windows** (разработка), целевой деплой: **Linux**.
- Учитывай кросс-платформенные нюансы (build tags, пути и т.д.).
- При предложении shell-команд давай варианты для PowerShell и bash.

## Чего НЕ делать

- Не генерировать код без объяснений.
- Не менять `.proto`-файлы без явной просьбы (перегенерация — отдельный шаг).
- Не предлагать внешние зависимости, если задачу решает стандартная библиотека.
- Не использовать ORM (предпочитаем `database/sql` + `pgx` / `sqlc`).
