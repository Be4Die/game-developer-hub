# Plan: Fix Incorrect Server Discovery Data

## Problem Statement
The method that provides connection data to clients (discovery) returns incorrect values:
- Hostname resolves to internal Docker DNS name (`gdh-node`) which is not reachable from clients.
- Port is `0` instead of the actual mapped host port.
Result: clients fail with `lookup gdh-node: no such host` and cannot connect.

## Root Causes

### 1. Incorrect Node External Address
The node announces itself using `external_address: "gdh-node:44044"` (from `game-server-node/config/docker.yaml`).  
This is the container's internal DNS name, not resolvable from outside Docker. The discovery service uses the host part of this address for client connections.

### 2. Dynamic Port Allocation Bug
When a game server instance requests a dynamic port (`HostPort = 0`), the Docker runtime incorrectly sets `HostPort: "0"` in the port binding. Docker interprets this as a literal port 0 (invalid) rather than "any", so it does not allocate a port and the subsequent `getPort()` call fails. The code logs a warning but still returns `actualPort = 0`, which gets stored in the instance record and later returned by discovery.

## Solution

### Phase 1 – Fix Docker Port Binding (Code Change)
**File:** `game-server-node/internal/infrastructure/runtime/docker/runtime.go`

- In `CreateContainer`, adjust port binding logic:
  - If `opts.HostPort` is `0`, set `HostPort` to empty string `""` (Docker's indicator for dynamic allocation). Otherwise set to the stringified port.
- After calling `getPort()`, treat an error or zero port as fatal: return the error instead of continuing with an invalid port. This prevents creation of broken instances.

### Phase 2 – Configure Node with a Client‑Reachable Address (Config Change)
**File:** `game-server-node/config/docker.yaml`

- Replace the `external_address` value:
  - Change `external_address: "gdh-node:44044"` to `external_address: "host.docker.internal:44044"`.
  - On Linux, use the Docker gateway IP such as `172.17.0.1:44044` or provide a LAN‑accessible hostname/IP.
- This makes the announced address resolvable both from the orchestrator container (via host gateway) and from clients running on the host machine.
- The host part (`host.docker.internal`) will be used by discovery; the port returned will be the actual game server host port (now correctly allocated).

### Phase 3 – Verification & Testing
1. Rebuild images (or restart services if using mounted configs).
2. Ensure node successfully re‑announces with the new address.
3. Start a new game server instance via the API.
4. Call the discovery endpoint (`/api/v1/discovery`) and verify the response contains a host like `host.docker.internal` (or your configured host) and a non‑zero port (e.g., `32768`).
5. From a browser or WebSocket client on the host, connect to `ws://<host>:<port>/ws` and confirm successful connection.

### Optional Improvements (Future Work)
- If needed, introduce a separate `client_address` field to decouple gRPC address from client‑facing address completely. This would require protobuf, DB, and service changes.
- Add a background job to clean up or fix existing instances that have `HostPort = 0`.

## Files to Modify
- `game-server-node/internal/infrastructure/runtime/docker/runtime.go` – port binding logic and error propagation.
- `game-server-node/config/docker.yaml` – set proper `external_address`.
- (Optional) migration/seed data to correct existing nodes.

After applying these changes, the discovery method will return valid IP/hostname and correct external ports, enabling clients to connect successfully.
