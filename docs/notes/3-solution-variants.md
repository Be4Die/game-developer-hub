# Варианты решения

## Ручное решение

Так как процесс на данным момент частично автоматизирован, рассмотрим гипотетический сценарий, в котором отсутствует какая-либо автоматизация — все операции выполняются вручную без использования CI/CD.

### Публикация игры

1. Разработчик собирает билд игры локально.
2. Разработчик отправляет архив с билдом в Telegram-чат.
3. Сотрудник платформы скачивает архив.
4. Сотрудник подключается к серверу по SSH.
5. Сотрудник вручную размещает файлы в нужной директории.
6. Сотрудник перезапускает веб-сервер или сервис.
7. Сотрудник сообщает в чат, что билд выложен на dev.
8. Модератор тестирует игру, пишет замечания в Telegram.
9. Разработчик вносит правки, повторяет шаги 1–7.
10. При положительном решении сотрудник вручную копирует файлы на prod-сервер.

### Управление игровым сервером

1. Разработчик собирает билд сервера локально.
2. Разработчик отправляет архив с билдом в Telegram-чат.
3. Сотрудник скачивает архив.
4. Сотрудник подключается к выделенному серверу по SSH.
5. Сотрудник размещает файлы, настраивает окружение.
6. Для запуска инстанса разработчик пишет запрос в чат.
7. Сотрудник подключается по SSH и запускает процесс вручную.
8. Сотрудник сообщает в чат, что сервер запущен.
9. Для остановки — аналогичная процедура через чат и SSH.
10. Логи сервера недоступны разработчику — при необходимости сотрудник копирует фрагменты логов в чат.

### Модерация

Процесс модерации при ручном решении идентичен текущему: вся коммуникация ведётся через Telegram-чаты, единый статус игры отсутствует, история замечаний не сохраняется в структурированном виде.

### Отчётность

В полностью ручном сценарии сотрудник платформы подключается к базам данных и сервисам напрямую, вручную выгружает числовые показатели — количество игроков, показы рекламы, доходы и прочее — сводит их в таблицу (например, в Excel или Google Sheets) и отправляет каждому разработчику в Telegram. При росте числа проектов этот процесс становится неуправляемым: на формирование и рассылку отчётов по всем играм может уходить несколько рабочих дней. Разработчик при этом не имеет никакого доступа к данным самостоятельно и полностью зависит от сотрудника платформы.

## Текущее решение (частичная автоматизация)

Текущий процесс описан в документе [2-problem.md](2-problem.md) и представляет собой частичную автоматизацию на базе GitHub Actions.

### Публикация игры

1. DevOps-инженер создаёт репозиторий на GitHub и настраивает CI/CD workflow.
2. Разработчик коммитит билд в репозиторий.
3. GitHub Actions автоматически разворачивает билд на dev-сервере.
4. Модератор тестирует игру, пишет замечания в Telegram.
5. Разработчик вносит правки, коммитит повторно.
6. При положительном решении DevOps-инженер вручную запускает экшен для деплоя на prod.

### Управление игровым сервером

1. DevOps-инженер создаёт отдельный репозиторий и workflow для серверной части.
2. Разработчик выгружает билд сервера через GitHub Actions.
3. Для запуска/остановки инстанса разработчик обращается через Telegram.
4. DevOps-инженер вручную запускает соответствующий экшен.
5. Логи сервера разработчику недоступны напрямую.

### Модерация

Процесс модерации не автоматизирован: коммуникация через Telegram, история замечаний разрозненна.

### Отчётность

Сбор данных частично автоматизирован: существует собственная десктоп-программа, которая обращается к API внутренних микросервисов платформы и формирует документ с отчётом. Однако процесс всё равно требует ручного участия менеджера — он запускает программу, дожидается результата и отправляет готовый отчёт каждому разработчику в Telegram-чат. Саму десктоп-программу необходимо поддерживать как отдельный продукт. Разработчик по-прежнему не имеет самостоятельного доступа к статистике и видит результаты только раз в месяц. При большом количестве проектов формирование и рассылка отчётов растягивается на несколько дней.

## Аналоги

### Серверные решения

#### Unity Game Server Hosting (Multiplay)

Облачный хостинг игровых серверов от Unity. Предоставляет автоматическое масштабирование, управление инстансами через веб-интерфейс, интеграцию с Unity-проектами.

- Ориентирован на десктопные и мобильные игры.
- Ценообразование основано на потреблении ресурсов — невыгодно для множества небольших веб-проектов.
- Не интегрируется с процессом публикации на сторонних площадках.

#### AWS GameLift

Сервис Amazon для развёртывания и масштабирования выделенных игровых серверов. Поддерживает матчмейкинг, автоскейлинг, мониторинг.

- Высокий порог входа: требует настройки инфраструктуры AWS.
- Ценообразование по потреблению — затратно для частых экспериментов.
- Заточен под высоконагруженные проекты, избыточен для веб-игр.

### Консоли разработчика

#### Steamworks

Консоль разработчика Steam. Позволяет загружать билды, управлять страницей игры, просматривать статистику, взаимодействовать с игроками.

- Проприетарная система, работает только в экосистеме Steam.
- Ориентирована на десктопные игры.
- Модель дистрибуции не применима к веб-играм.

#### itch.io Dashboard

Панель управления для публикации инди-игр. Простой интерфейс загрузки билдов, настройка страницы, аналитика продаж.

- Не предоставляет серверный хостинг.
- Ориентирована на одиночные/локальные игры.
- Отсутствует модерационный workflow — игры публикуются напрямую.

## Вывод

Ручное решение не масштабируется и создаёт критическую зависимость от доступности сотрудников. Текущее решение с частичной автоматизацией снимает часть нагрузки, но сохраняет ключевые проблемы: ручная настройка CI/CD под каждый проект, отсутствие самостоятельного доступа разработчика к серверам, разрозненная коммуникация в мессенджерах, а также отсутствие оперативного доступа к статистике — разработчик узнаёт о результатах своей игры только раз в месяц из отчёта, который ему присылает менеджер.

Существующие аналоги на рынке не решают задачу комплексно:
- **Серверные решения (Multiplay, GameLift)** экономически невыгодны для веб-игр и не интегрируются с процессом публикации.
- **Консоли разработчика (Steamworks, itch.io)** являются проприетарными, привязаны к своим площадкам и не предоставляют инструментов управления серверами. Встроенная аналитика Steamworks привязана к экосистеме Steam и неприменима к сторонним платформам.

Ни одно из рассмотренных решений не покрывает полный цикл «публикация + модерация + оркестрация серверов + аналитика и отчётность» для платформы веб-игр, что обосновывает необходимость проектирования собственного инструмента.
