# Анализ проблемы

## Формулировка проблемы

У платформы для публикации веб-игр отсутствует собственный инструментарий для взаимодействия с разработчиками и управления процессом публикации. Все ключевые операции — деплой, управление серверами, коммуникация, формирование отчётности — выполняются вручную через сторонние сервисы или отдельные утилиты, что создаёт значительные издержки по времени и ресурсам, а также снижает качество процессов по мере роста количества проектов.

## Текущее состояние (As-Is)

Текущие процессы описаны в нотации IDEF0 и разделены на два слабо связанных контура: публикация игры и управление игровым сервером.

### Процесс 1: Публикация игры на платформе

#### Контекстная диаграмма

![Публикация игры — контекст](./compiled-images/publish-game-as-is-context.svg)

#### Декомпозиция

![Публикация игры — декомпозиция](./compiled-images/publish-game-as-is-decompose.svg)

**Описание процесса:**

1. **Настроить CI/CD.** DevOps-инженер вручную создаёт репозиторий на GitHub и пишет workflow для конкретной игры. Конфигурация основана на практиках боевых бекенд-проектов, что избыточно для сущности «игра».
2. **Загрузить билд в dev.** Разработчик коммитит билд, GitHub Actions разворачивает его в dev-окружении.
3. **Проверить игру.** Модератор тестирует игру на соответствие требованиям платформы.
4. **Согласовать публикацию.** Модератор и разработчик обмениваются замечаниями через Telegram. На этом этапе разработчик также предоставляет SEO-данные и медиа-контент.
5. **Выгрузить на прод.** При положительном решении DevOps-инженер вручную запускает экшен для деплоя на продакшен.
6. **Сформировать отчёт.** После публикации игра начинает накапливать статистику. В конце месяца менеджер (он же модератор) запускает десктоп-программу, которая обращается к API внутренних микросервисов платформы, собирает данные по каждой игре — количество игроков, показы рекламы, среднее время сессии, возвраты и другие метрики — и формирует документ-отчёт. Готовый отчёт менеджер вручную отправляет разработчику в Telegram-чат. Весь месяц до получения отчёта разработчик не имеет доступа к статистике своей игры.

### Процесс 2: Управление игровым сервером

#### Контекстная диаграмма

![Управление сервером — контекст](./compiled-images/manage-server-as-is-context.svg)

#### Декомпозиция

![Управление сервером — декомпозиция](./compiled-images/manage-server-as-is-decompose.svg)

**Описание процесса:**

1. **Настроить CI/CD сервера.** DevOps-инженер создаёт отдельный репозиторий и workflow для серверной части игры.
2. **Загрузить билд сервера.** Разработчик выгружает серверный билд через GitHub Actions на выделенный сервер.
3. **Запустить инстанс.** Разработчик не может самостоятельно запустить сервер — необходимо запросить запуск через Telegram, после чего DevOps запускает экшен вручную.
4. **Остановить инстанс.** Аналогично: для остановки сервера разработчик обращается через Telegram к DevOps-инженеру.

## Недостатки текущего процесса

| Метрика | Проявление |
|---|---|
| **Время** | Публикация одной игры требует ручной настройки репозитория, workflow и координации между несколькими участниками. Каждое действие с сервером — отдельный запрос через Telegram и ручной запуск экшена. Формирование отчётов занимает несколько дней в конце месяца из-за необходимости обработать все проекты вручную, при большой нагрузке рассылка растягивается. |
| **Ресурсы** | DevOps-инженер задействован для рутинных операций. Менеджеры выступают посредниками между разработчиком и модератором. Менеджер тратит время на ручной запуск десктоп-программы и рассылку отчётов каждому разработчику. Десктоп-программу необходимо поддерживать как отдельный продукт. Процессы не масштабируются при увеличении количества проектов. |
| **Качество** | Отсутствует прозрачность: нет единого статуса игры, нет истории модерации, обратная связь теряется в мессенджерах. Разработчик не видит логов сервера. Разработчик не имеет доступа к статистике игры в течение месяца — возникает эффект «чёрного ящика», когда невозможно оценить, как живёт опубликованный проект, до получения отчёта. |
| **Отсутствие функций** | Разработчик не имеет самостоятельного доступа к управлению проектом: не может загрузить билд, запустить/остановить сервер, посмотреть логи, отследить статус модерации, просмотреть статистику и финансовые показатели игры через единый интерфейс. |

## Вывод

Текущий процесс публикации и сопровождения игр на платформе не масштабируется, требует значительных ручных усилий и не обеспечивает приемлемый пользовательский опыт ни для разработчиков, ни для команды модерации. Отдельной проблемой является отсутствие оперативного доступа к аналитике — разработчик узнаёт о результатах своей игры только раз в месяц, что не позволяет ему своевременно реагировать на изменения и принимать решения по развитию проекта. Готовые рыночные решения не применимы: серверные платформы экономически невыгодны для веб-игр, а консоли разработчика являются проприетарными продуктами, привязанными к чужим площадкам и моделям взаимодействия. Это обосновывает необходимость проектирования собственного инструмента.
