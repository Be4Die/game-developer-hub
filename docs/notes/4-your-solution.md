# Целевое решение (TO-BE)

## Концепция

**Game Developer Hub** — единая веб-платформа, объединяющая полный цикл публикации игры, управления игровыми серверами и доступа к аналитике. Система заменяет ручные процессы и разрозненные инструменты единым интерфейсом для разработчиков и модераторов.

Ключевой принцип: **разработчик выполняет все операции самостоятельно**, без привлечения DevOps-инженера или менеджера, и имеет постоянный доступ к данным о своих проектах.

#### Декомпозиция

#### Описание процесса

| Шаг | Действие | Исполнитель |
|-----|----------|-------------|
| 1 | **Зарегистрироваться.** Разработчик самостоятельно регистрируется на платформе и получает аккаунт. | Разработчик |
| 2 | **Создать черновик игры.** Разработчик создаёт новый проект в интерфейсе хаба. | Разработчик |
| 3 | **Загрузить билд.** Разработчик загружает архив с билдом (zip, tar.gz) через веб-интерфейс. | Разработчик |
| 4 | **Развернуть на dev.** По команде «Сохранить черновик» система автоматически разворачивает билд в dev-окружении. | Система |
| 5 | **Заполнить данные игры.** Разработчик заполняет SEO-данные, описание, загружает медиа-контент. | Разработчик |
| 6 | **Отправить на модерацию.** Разработчик нажимает кнопку «Отправить на модерацию», создаётся тикет. | Разработчик |
| 7 | **Провести модерацию.** Модератор видит тикет в своём интерфейсе, тестирует игру, выносит вердикт. При отклонении — пишет замечания во встроенном чате. | Модератор |
| 8 | **Опубликовать на prod.** При положительном вердикте система автоматически публикует игру на prod. | Система |
| 9 | **Просмотреть статистику.** После публикации разработчик получает доступ к разделу аналитики в интерфейсе хаба. Система автоматически подтягивает данные по API внутренних микросервисов платформы с периодичностью раз в день. Разработчик видит графики динамики по ключевым метрикам — количество игроков, показы рекламы, среднее время сессии, возвраты и другие показатели — и может скачать сформированный отчёт за выбранный период. | Разработчик / Система |

### Процесс 2: Управление игровым сервером

#### Контекстная диаграмма

![Управление сервером TO-BE — контекст](./compiled-images/manage-server-to-be-context.svg)

#### Декомпозиция

![Управление сервером TO-BE — декомпозиция](./compiled-images/manage-server-to-be-decompose.svg)

#### Описание процесса

| Шаг | Действие | Исполнитель |
|-----|----------|-------------|
| 1 | **Загрузить билд сервера.** В разделе «Онлайн» черновика игры разработчик загружает архив серверного билда. Система хранит до 10 последних версий. | Разработчик |
| 2 | **Настроить режим работы.** Разработчик выбирает режим: автоматический (авторестарт, масштабирование при нагрузке) или ручной (для отладки). | Разработчик |
| 3 | **Запустить инстанс.** В ручном режиме — по кнопке в интерфейсе. В автоматическом — система управляет инстансами самостоятельно. | Разработчик / Система |
| 4 | **Остановить инстанс.** Разработчик останавливает инстанс через интерфейс (в ручном режиме). | Разработчик |
| 5 | **Просмотреть логи.** Разработчик просматривает логи работающего сервера в реальном времени. | Разработчик |
| 6 | **Получить коннект.** Клиент игры отправляет запрос в сервис дискавери и получает коннект (IP, порт) или ошибку. | Система |
| 7 | **Подключить ноду.** Разработчик устанавливает slave-агент на свою машину. Нода входит в режим вещания и появляется в списке. Разработчик выбирает ноду и авторизует её паролем (или вводит IP вручную). | Разработчик |

## Ключевые изменения относительно As-Is

| Аспект | As-Is | TO-BE |
|--------|-------|-------|
| **Создание проекта** | DevOps вручную создаёт репозиторий и workflow | Разработчик создаёт черновик в интерфейсе за минуту |
| **Загрузка билда** | Коммит в GitHub-репозиторий | Drag & drop архива в веб-интерфейс |
| **Деплой на dev** | GitHub Actions | Автоматически по команде «Сохранить черновик» |
| **Деплой на prod** | Ручной запуск экшена DevOps-инженером | Автоматически при одобрении модератором |
| **Коммуникация** | Telegram-чаты, сообщения теряются | Встроенный чат в тикете модерации, история сохраняется |
| **Управление сервером** | Запрос через Telegram → DevOps запускает экшен | Кнопки в интерфейсе, разработчик управляет сам |
| **Доступ к логам** | Через DevOps-инженера | В реальном времени в интерфейсе |
| **Статистика и отчётность** | Десктоп-программа + ручная рассылка менеджером раз в месяц | Автоматический дашборд с графиками в интерфейсе, обновление раз в день, скачивание отчётов за любой период |
| **Участие DevOps** | На каждом шаге | Только настройка инфраструктуры платформы |
| **Участие менеджера в отчётности** | Формирует и рассылает отчёты вручную | Не требуется — данные доступны разработчику автоматически |

## Вывод

Целевое решение устраняет ключевые проблемы текущего процесса:

- **Самостоятельность разработчика.** Все операции — от создания проекта до управления серверами и просмотра статистики — выполняются без привлечения сотрудников платформы.
- **Прозрачность.** Единый интерфейс с историей изменений, статусами, встроенной коммуникацией. Разработчик видит, как живёт его игра на платформе, в любой момент — не дожидаясь ежемесячного отчёта.
- **Масштабируемость.** Добавление новых проектов не требует ручной работы DevOps-инженера. Формирование отчётов происходит автоматически и не зависит от количества проектов на платформе.
- **Сокращение времени.** Публикация игры занимает минуты вместо часов/дней. Статистика доступна в реальном времени с периодичностью обновления раз в день.

Решение учитывает специфику веб-игр и модель отношений «платформа — разработчик», что отличает его от существующих аналогов на рынке.
