# Идея: Game Developer Hub

## Суть продукта

**Game Developer Hub** — веб-платформа, которая предоставляет единое рабочее пространство для всех участников процесса публикации игр: от загрузки проекта и его модерации до управления серверной инфраструктурой онлайн-игр и отслеживания статистики по опубликованным проектам.

Продукт объединяет в себе инструменты для размещения игр на площадке, взаимодействия между разработчиками и командой модерации, управления игровыми серверами онлайн игр, а также предоставляет разработчикам доступ к аналитике и финансовой отчётности — заменяя набор разрозненных сервисов и ручных процессов одним интерфейсом.

## Мотивация

Идея сформировалась из опыта работы в сфере разработки игр и непосредственного участия в запуске новой платформы для публикации веб-игр с фокусом на онлайн-проекты.

Изначально существовали две отдельные идеи:

- **Инструмент управления игровыми серверами** — удобный интерфейс для поднятия, отладки и деплоя серверов онлайн-игр без необходимости привлекать DevOps-специалиста на каждое действие.
- **Личный кабинет разработчика** — внутренний сервис платформы, через который разработчики публикуют игры, отслеживают статус модерации и взаимодействуют с командой площадки.

Обе идеи были объединены в единый хаб, поскольку решают задачи одного процесса — вывод игры на платформу и её дальнейшее сопровождение. Важной частью сопровождения является финансовая и продуктовая отчётность: разработчик должен понимать, как живёт его игра на платформе — сколько игроков приходит, какова динамика показов рекламы, сколько он заработал за период. Сейчас эта информация доставляется вручную раз в месяц через Telegram, что не даёт разработчику оперативно реагировать на изменения и создаёт эффект «чёрного ящика».

## Проблема

На данный момент у платформы отсутствуют собственные инструменты для обеспечения пользовательского опыта разработчиков и модераторов. Текущий процесс выстроен на сторонних сервисах и ручных операциях:

- **CI/CD через GitHub:** под каждую игру вручную заводится репозиторий и пишется workflow для выгрузки в два окружения (dev и prod). Конфигурацию настраивал DevOps-инженер, ориентируясь на практики боевых бекенд-проектов — это избыточно для сущности «игра».
- **Управление серверами через GitHub Actions:** разработчик не может самостоятельно остановить или перезапустить сервер — приходится вручную запускать экшены в интерфейсе GitHub.
- **Коммуникация через Telegram:** ведутся десятки чатов с разработчиками игр, задействовано множество участников, сообщения теряются, задачи остаются без внимания.
- **Отчётность через десктоп-программу:** в конце месяца менеджер запускает отдельную десктоп-программу, которая обращается к API внутренних микросервисов платформы, собирает статистику и формирует документ. Готовый отчёт отправляется разработчику в Telegram-чат. На протяжении всего месяца разработчик не имеет доступа к данным о своей игре и не видит динамику показателей.

## Основные сценарии

### 0. Регистрация
Регистрация и выдача аккаунта студии или отдельному разработчику на платформе.

### 1. Публикация игры
1. Разработчик создаёт черновик игры.
2. Загружает билд.
3. Билд разворачивается в dev-окружении.
4. Разработчик тестирует, как игра ведёт себя на платформе.
5. Заполняет SEO-данные, описание, медиа-контент.
6. Отправляет черновик на модерацию.
7. Модератор проверяет игру на соответствие требованиям платформы.
8. **Принято** → игра публикуется на prod.
9. **Отклонено** → модератор описывает замечания во внутреннем чате.
10. После публикации разработчик получает доступ к статистике и отчётам по игре.

### 2. Оркестрация серверов
Платформа предоставляет серверные мощности для онлайн-проектов. В черновике игры разработчик переходит в раздел «Онлайн», где может:

- Загрузить билд сервера.
- Поднять инстанс сервера.
- Просматривать логи в реальном времени.
- Протестировать авто-обнаружение серверов на клиенте.
- Подключить собственные серверы (ноды) через slave-микросервис, установленный на своей машине.

### 3. Отчётность и статистика
После публикации игры на prod разработчик получает доступ к разделу аналитики в интерфейсе хаба. Статистика обновляется с периодичностью раз в день и включает продуктовые и финансовые метрики: количество игроков, показы рекламы, среднее время сессии, количество возвратов в игру и другие показатели. Разработчик может просматривать графики динамики и скачивать сформированный отчёт за выбранный период. Данные подтягиваются автоматически по API внутренних микросервисов платформы — ручное формирование и рассылка отчётов больше не требуются.

## Целевые пользователи

- **Разработчики / студии веб-игр** — публикуют и сопровождают свои проекты на платформе, отслеживают статистику и финансовые результаты.
- **Модераторы платформы** — проверяют игры перед публикацией, взаимодействуют с разработчиками.

## Вывод

Идея направлена на решение реальной проблемы — отсутствие удобных инструментов для работы с платформой как со стороны разработчиков, так и со стороны команды модерации. Продукт представляет интерес с технической точки зрения: оркестрация игровых серверов, автоматизация CI/CD-процессов, встроенная аналитика с финансовой отчётностью и проектирование интерфейса для нескольких ролей пользователей. Основная цель — повысить качество работы площадки, сократить ручной труд и выстроить прозрачный процесс публикации и сопровождения игр.
