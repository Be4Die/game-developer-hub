# Правила оформления комментариев 

## Главное требование
Все комментарии пишем на русском языке. Исключения: устоявшиеся технические термины (nil, ctx, mutex, goroutine, timeout), имена ошибок, ссылки на внешние системы и ключевые слова линтеров.

## Базовая формула
`// [Имя] [глагол в настоящем времени] [объект] [при/для/возвращает] [условие или гарантия].`

Пример:
`// GetUser загружает профиль пользователя по ID из кэша. Возвращает ErrNotFound при отсутствии.`

## Что и где писать
- **Пакет.** Один комментарий на пакет. Описывает, зачем пакет существует и для какой задачи используется. Обычно кладется в `doc.go` или файл с именем пакета.
- **Экспортированные типы, функции, методы.** Начинаются строго с имени. Описывают контракт: что делает, что ожидает, что возвращает. Не описываем внутреннюю реализацию.
- **Внутренние (неэкспортированные).** Пишем только если код неочевиден. Объясняем «почему так», а не «что делает». Используем пометки NOTE, TODO, HACK со ссылкой на задачу или коммит.

## Обязательно указывать для публичного API
- Потокобезопасность: безопасен для конкурентного использования или нет.
- Нулевое значение: можно ли использовать пустой struct или нужно вызывать конструктор.
- Ошибки: какие конкретные ошибки возвращаются и при каких условиях.

## Чего избегать
- Дублирование имени. Плохо: `GetUser получает пользователя`. Хорошо: `GetUser загружает данные из внешнего сервиса с повторными попытками`.
- Размытые формулировки. Плохо: `возвращает ошибку при сбое`. Хорошо: `возвращает ErrTimeout, если запрос не уложился в 5 секунд`.
- TODO без контекста. Плохо: `// TODO: переделать`. Хорошо: `// TODO(#142): перейти на v2 API после релиза бэкенда`.
- Пропущенная точка в конце предложения или строчная буква в начале. `go doc` и линтеры на это реагируют.

## Практики крупных компаний
В Google, Uber, Cloudflare и HashiCorp принято следующее:
- Комментарий к публичному идентификатору это часть контракта API. Если поведения нет в комментарии, его не гарантируют.
- Документируем интерфейс, а не реализацию. Код может меняться, контракт остается.
- Все экспортированные сущности должны иметь комментарии. Пропуски считаются техдолгом и блокируют мерж.
- Внутренние комментарии оставляют только для нетривиальных решений, обходных путей или известных ограничений.

## Автоматическая проверка
В CI используем `golangci-lint` с включенными правилами `revive` и `godot`. Они проверяют наличие комментариев, заглавную букву в начале и точку в конце.
Локально рендеринг можно проверить командой `go doc -all ./...`.

## Шаблон для копирования
```go
// Package [имя] реализует [краткое описание ответственности] для [задачи/домена].
package [имя]

// [Тип] управляет [ресурсом] через [механизм].
// Безопасен для конкурентного использования. Возвращает ErrX при Y.
// Нулевое значение корректно и означает режим по умолчанию.
type [Тип] struct { ... }

// [Метод] выполняет [действие] для [объекта].
// Требует валидного ctx. Не безопасен во время инициализации.
func (t *[Тип]) [Метод](ctx context.Context) ([результат], error) { ... }
```

Если `go doc` выводит ваш комментарий понятно и без лишних деталей, значит он соответствует стандарту.
