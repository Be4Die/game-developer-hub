# Правила оформления коммитов в Git

## Главное требование
Сообщения коммитов пишем на русском языке. Типы коммитов, технические термины, имена веток и ссылки на трекеры остаются на английском. Коммит это запись в истории проекта, а не личная заметка. Пишем так, чтобы через год любой участник команды понял контекст без открытия диффа.

## Базовая формула
`<type>(<scope>): <subject>`

Пример: `feat(auth): добавить проверку срока жизни токена`

Если нужен контекст или пояснение, добавляем тело и футер через пустую строку:
```
<type>(<scope>): <subject>

<опциональное тело>

<опциональный футер>
```

## Из чего состоит сообщение
- **Тип (type).** Определяет характер изменения. Используем стандарт Conventional Commits:
  `feat` – новая функциональность
  `fix` – исправление бага
  `docs` – документация
  `style` – форматирование, отступы, линтеры (без изменения логики)
  `refactor` – изменение кода без изменения поведения
  `perf` – оптимизация производительности
  `test` – тесты
  `chore` – сборка, зависимости, конфиги, рутина
  `ci` – изменения в CI/CD
  `build` – системы сборки
  `revert` – откат предыдущего коммита
- **Область (scope).** Опционально. Указывает модуль, пакет или компонент, который затронут. Пишем в скобках без пробелов: `(auth)`, `(db)`, `(api)`.
- **Тема (subject).** Краткое описание изменения. До 50 символов. Глагол в повелительном наклонении. Без точки в конце. Заглавная буква в начале.
- **Тело (body).** Пишем только если заголовок не раскрывает контекст. Объясняем *почему* и *что изменилось*, а не *как*. Разделяем заголовок и тело одной пустой строкой. Переносим длинные строки на 72 символа.
- **Футер (footer).** Ссылки на задачи, breaking changes, соавторы. Форматы: `Closes #123`, `Refs: JIRA-456`, `BREAKING CHANGE: описание`.

## Обязательные правила
- Один коммит = одна логическая задача. Не смешиваем рефакторинг, фиксы и новые фичи.
- Всегда используем повелительное наклонение: `добавить`, `исправить`, `удалить`, `обновить`. Не `добавил`, `фиксит`, `поправил`.
- Breaking changes маркируем явно в футере: `BREAKING CHANGE: изменена сигнатура метода, требуется миграция конфигов`.
- Если коммит решает задачу из трекера, ссылка обязана быть в футере.

## Чего избегать
- Общие формулировки: `fix bug`, `update`, `wip`, `small changes`, `refactor code`.
- Описание реализации вместо причины: вместо `поменять цикл на мапу` пишем `ускорить обработку записей за счет хеш-таблицы`.
- Пропущенные пустые строки между заголовком и телом.
- Точка в конце заголовка.
- Слияние несвязанных файлов в один коммит (например, правки линтера и новая бизнес-логика).

## Практики крупных компаний
- **Google.** Описания изменений требуют четкого контекста и ссылки на дизайн-документ или RFC. Явно указывают, какие сервисы или тесты затронуты. Коммит без контекста не проходит ревью.
- **Uber / Netflix.** Строгий Conventional Commits. На основе истории автоматически генерируется changelog и рассчитывается семантическая версия. Отклонение от формата блокирует пайплайн.
- **HashiCorp / Cloudflare.** Футеры обязательны для связки с трекером. Breaking changes сопровождаются инструкциями по миграции. Коммиты squash-ятся в main, но история промежуточных веток должна оставаться чистой.
- **Общее правило индустрии.** Коммит проверяется так: `git log --oneline` должен давать понятную картину релиза без открытия кода.

## Автоматическая проверка
В локальной разработке и CI используем хуки и линтеры:
- `husky` + `commitlint` – валидация формата перед созданием коммита
- `standard-version` или `semantic-release` – автоматический changelog и версионирование
- `git log --pretty=format:"%h %s"` – быстрая проверка истории

Пример конфигурации commitlint:
```
module.exports = {extends: ['@commitlint/config-conventional']}
```

## Шаблон для копирования
```
feat(api): добавить пагинацию к эндпоинту /users

Раньше список отдавался целиком, что вызывало таймауты при >10k записей.
Теперь используется limit/offset с дефолтом 50. Максимум — 200.

Closes #142
```

Если `git log --oneline` показывает последовательность осмысленных действий, а не хаотичные правки, значит стандарт соблюдается.
